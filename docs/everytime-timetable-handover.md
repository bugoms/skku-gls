# 인수인계 — 에타(에브리타임) 시간표 연동 (GLS 확장앱)

> 이 메모만 읽고 바로 이어서 작업할 수 있게 정리. 대상: 다른 Claude 세션/코워커.
> 작성 2026-07-19. 근거 = 실제 코드 + `docs/api-notes.md` + 대화 결정.

## 0. 목표 (한 줄)
GLS 확장앱에서 만든 **내 시간표를 에타(everytime.kr) 시간표에 등록**.
**방식(확정): 에타에서 강의를 "검색" → "시간표에 추가"를 눌러 등록** = **실제 에타 강의로 등록**(취지: 에타 강의평·공유 등 기능과 연결됨). 매칭 안 되는 과목만 "직접 추가(커스텀)"로 폴백.
현재 **미구현**. **리버스엔지니어링용 캡처가 선행 과제** → 캡처 받은 뒤 구현.

## 2. 접근 (확정)
### 주 경로 ⭐ — 강의 검색 → 시간표에 추가
- GLS 과목을 **에타 강의 검색**으로 찾아, 그 **강의 id(primaryKey)** 로 **시간표에 추가**.
- 장점: 에타의 **실제 강의로 등록** → 강의평/공유/친구시간표 등과 연결(취지에 부합). 시간·장소는 에타가 강의 DB에서 자동으로 채움(우리가 시간 인코딩 안 보내도 됨).
- **핵심 난제 = GLS 과목 → 에타 강의 매칭.** 과목명+교수(+시간)로 후보 중 하나 선택.
  - 이미 만든 **gap 매칭 로직**(`src/content/everytime-link.js`의 과목명↔교수 간격 기반, 논리회로설계 vs 논리회로설계실험 구분)을 재사용/이식. 여기선 **시간(요일·교시)** 까지 있어 분반 특정에 더 유리.

### 폴백 — 직접 추가(커스텀 강의)
- 에타 DB에 없거나(개설 차이) 매칭 실패(외국인 교수 표기차 등) 과목은 **직접 추가**(과목명·교수·요일/시간·장소 직접 입력)로 등록. 실패 과목은 사용자에게 목록으로 알려주고 이 경로 제안.

## 3. 선행 과제 — 사용자에게 요청할 캡처 (레시피)
에타는 Nexacro가 아니라 일반 웹 → **콘솔 후킹 불필요, DevTools Network 캡처면 충분.**

**준비**: 에타 로그인 → 시간표 메뉴 → **테스트용 시간표 새로 생성** → **2026년 2학기**로 설정. 끝나면 테스트 시간표 삭제.

**공통 세팅**: F12 → Network → **Preserve log 체크** → 필터 Fetch/XHR → 각 동작 직전 Clear.

**캡처 (A) 강의 검색**
1. 시간표에서 **"+ 강의 추가"** → 검색창에 실제 개설 과목(예: `일반물리학2` 또는 본인 수강 과목) 입력해 **검색**.
2. 뜬 **검색 요청(POST)** → 우클릭 **Copy as cURL (bash)** + **Response** 본문 복사.

**캡처 (B) 시간표에 추가**
3. 검색 결과에서 한 강의의 **"추가"(시간표에 추가)** 클릭.
4. 그때 뜬 **추가 요청(POST)** → **Copy as cURL** + **Response** 복사.

**캡처 (C, 폴백용·선택) 직접 추가**
5. "직접 추가"로 `테스트과목`/`홍길동`/`월 10:00~11:00`/`테스트강의실` 저장 → 그 요청 cURL + Response.

**민감정보 제거(값만 가림, 키/구조는 유지)**: `Cookie:` 값 → `<REDACTED>`, 토큰 헤더/필드(`x-csrf-token`·`authorization`·`_token` 등)는 **키 이름 남기고 값만** `<REDACTED>`. 학번 등 개인정보도 가림.

**받아야 할 것 체크리스트**:
- [ ] (A) 강의 검색 요청 cURL + Response (Response에 **강의 id/primaryKey·과목명·교수·시간·분반** 있는지 확인)
- [ ] (B) 시간표 추가 요청 cURL + Response (**어떤 값으로 강의를 지정**하는지: id? primaryKey? / **시간표 id**·학기 파라미터)
- [ ] (C, 선택) 직접 추가 요청 cURL + Response
- [ ] 요청 호스트가 `everytime.kr`인지 **`api.everytime.kr`**인지 (권한 추가 판단)

## 4. 캡처에서 뽑아낼 것 (구현 입력값)
- **검색 API**: 엔드포인트·파라미터(키워드·학기·캠퍼스?), 응답의 **강의 식별자(id/primaryKey)** + 매칭용 필드(과목명·교수·요일/시간·분반).
- **추가 API**: 엔드포인트·파라미터(**시간표 id**, **강의 식별자**, 학기), 성공 응답 포맷.
- **CSRF/토큰**: 이름 + **어디서 얻는지**(쿠키/이전 요청/페이지 전역). 재현에 필요.
- **시간표 id·학기(year/term) 인코딩**: 우리 `yearTerm`(예 `2026-20`, 20=2학기; api-notes §4) → 에타 값 매핑.
- (폴백용) 직접 추가의 요일/시간 인코딩·필드.

## 5. 구현 계획 (캡처 후)
1. **매니페스트**: 에타 **시간표 페이지**에 content script 추가(현재는 `everytime.kr/lecture/*` 만 → 시간표 URL 패턴 추가). API가 `api.everytime.kr`면 `host_permissions`에 `https://api.everytime.kr/*` 추가.
2. **통신 컨텍스트**: 에타 일반 웹 → GLS의 MAIN world/브릿지 **불필요**. 단 **세션 쿠키 + CSRF** 필요 →
   - 권장: **에타 오리진 content script(ISOLATED)에서 same-origin `fetch`**(세션 자동 포함), 또는 background `fetch`(host_permissions로 쿠키 포함). CSRF 소싱 방식은 **캡처로 확정.**
3. **핵심 로직 (과목마다 반복)**:
   1. 에타 **강의 검색**(과목명 키워드, 2026 2학기).
   2. 후보 중 **매칭**: 과목명 정확일치 + 교수 일치(+ 요일/시간 일치)로 유일 후보 선택. → `everytime-link.js`의 gap 매칭 재사용, **시간 비교 추가**로 분반 특정.
   3. 매칭된 강의의 **id로 시간표에 추가**.
   4. 매칭 0/애매/시간불일치 → **실패 목록**에 담아 사용자에게 표시(+ 폴백: 직접 추가 제안).
4. **데이터 소스**: 활성 시간표 과목 = `chrome.storage.local` 의 **`gls_tables` = { tables:[{id,name,courses:[course...]}], activeId }`**. content.js 런타임엔 `tables`/`activeId`/`myTable`(활성 courses). `course` 필드: name, professor, code, section, codeSection, schedule, campus, yearTerm, year, term, credits, _color, id. 시간 파싱 = `src/lib/schedule.js` `SCHED.parseSchedule(course.schedule)` → `[{day,startMin,endMin}]`.
5. **UI**: `src/content/content.js` 시간표 헤더(rchead)에 **"에타로 내보내기"** 버튼(현재 활성 시간표 기준) → **확인창**(쓰기 동작) → 진행/결과 토스트(성공 n개·실패 목록). 온라인/시간없는 과목은 스킵.

## 6. 재사용할 코드 / 패턴
- **`src/content/everytime-link.js`**: 과목명↔교수 **gap 매칭 로직** — 강의 검색 결과 매칭에 그대로 활용(+ 시간 비교 추가). 에타 오리진 실행 컨텍스트 참고.
- **`src/lib/schedule.js`**: `SCHED.parseSchedule` / `assignLanes`. GLS 시간 파싱 재사용(매칭 시 요일/시간 비교, 폴백 직접추가 시 시간 변환).
- **`docs/api-notes.md` §8 + `src/page-bridge/bag-bridge.js`**: 쓰기 API 리버스엔지니어링/재현 참고. (에타는 Nexacro 아님 → 암호화·8인자 재생 불필요, 평범한 `fetch`.)
- **`src/content/content.js`**: 시간표 데이터 모델·다중 시간표 로직(activeTable/setActive…)·토스트/확인 UI.

## 7. 주의사항 (유지 조건)
- **빌드 도구 금지** — 순수 JS·무빌드.
- **쓰기 동작 → 실행 전 확인 UI 필수** (사용자 에타 시간표를 바꿈). 매칭 애매하면 자동등록 말고 사용자 확인.
- **세션 쿠키/CSRF 토큰은 민감** — 로컬만, 외부 공유·로그 금지. 캡처도 값은 가리고 받기.
- **에타 ToS/봇 정책 유의** — 본인 계정·저빈도·클릭당으로 제한. 자동 대량 등록 지양. 내부 API 비공개라 개편 시 파손 가능(취약).
- **어시스턴트 샌드박스에서 사용자 에타 계정 쓰기 실행 금지** — 코드 작성/읽기 재현만. **실제 등록 동작은 사용자가 브라우저에서 검증.**
- 수정 후 `node --check` (+ 파서 건드리면 `node tests/parser.test.js`). 확장앱 새로고침+F5.
- **커밋 대기 중**: §1의 미커밋 변경은 사용자 검증 후 커밋 예정. 이 연동도 검증 전까지 커밋하지 말 것(사용자 지시).
