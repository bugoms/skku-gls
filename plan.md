# SKKU GLS 과목 위치 찾기 확장앱 — 상세 계획 (plan.md)

> **구현 상태 (2026-07-17):** Phase 1(파서) + Phase 2(응답 후킹·로컬 인덱스·검색 UI) **구현 완료**. 로드 방법·사용법은 [README.md](README.md). 파서는 실제 fixture 24개 케이스로 검증됨(`node tests/parser.test.js`). 남은 것: 자동 영역 이동(Nexacro 폼 이름 확정 후), 자동 전체수집.

## 1. 배경 및 문제 정의

성균관대 GLS 전자시간표에서 과목을 책가방에 담으려면, 먼저 그 과목이 **어느 메뉴(학사-전공과목 / 학사-교양·기타과목 / 학사-DS과목 …)의 어느 영역(성균인성·리더십, 글로벌, 인문사회과학/자연과학기반, 자연/과학/기술 …)에 분류되어 있는지**를 알아야 한다.

문제는 이 분류가 직관적이지 않다는 것:

- 예: "일반물리학2"가 `학사-교양/기타과목 > 인문사회과학/자연과학기반`에 있는지, `자연/과학/기술`에 있는지, 아니면 `학사-전공과목`에 있는지 알 수 없어 메뉴를 하나씩 눌러가며 찾아야 함.
- 영역 개수가 13개 이상이고, 캠퍼스 구분(인문사회 / 자연과학 / i-Campus)까지 곱해지면 탐색 경우의 수가 매우 많음.
- 2025학번부터 영역 분류 체계가 바뀌어(화면 하단 경고문 참고) 학번에 따라 같은 과목도 다른 영역으로 보일 수 있음 → 혼란 가중.

**해결하려는 것:** 과목명(또는 학수번호)을 검색하면 → "이 과목은 `학사-교양/기타과목 > 자연/과학/기술` (자연과학캠퍼스)에 있습니다"처럼 **정확한 메뉴 경로**를 알려주고, 가능하면 그 위치로 바로 이동시켜 주는 브라우저 확장앱.

## 2. 목표 / 비목표

### 목표 (v1)
1. GLS 전자시간표 화면 위에서 과목명/학수번호로 검색.
2. 검색 결과에 **메뉴 경로**(예: `학사-교양/기타과목 → 글로벌`)와 과목 기본 정보(학수번호-분반, 교수, 학점, 캠퍼스, 시간)를 표시.
3. 결과 클릭 시 해당 메뉴/영역 탭으로 **자동 이동**(가능한 경우).

### 목표 (v2 이후, 선택)
- 초성 검색·부분 일치·오타 허용 검색.
- 학기 전체 과목 로컬 캐싱으로 즉시 검색.
- "책가방 바로 담기" 원클릭 버튼.

### 비목표
- 수강신청(sugang.skku.edu) 자체의 자동화 — 매크로로 오해될 소지가 있고 학교 정책 위반 위험. **하지 않음.**
- 로그인 자동화 / 세션 관리 — 사용자가 GLS에 이미 로그인한 상태를 전제.
- 성적·개인정보 등 전자시간표 외 GLS 기능 접근.

## 3. 사용자 시나리오

1. 학생이 GLS에 로그인하고 전자시간표 화면을 연다.
2. 확장앱이 화면 상단(또는 우측)에 검색바를 삽입한다.
3. "일반물리학" 입력 → 엔터.
4. 결과 목록:
   ```
   일반물리학2 (GEDB005-41) | 홍길동 | 3학점 | 자연과학캠퍼스
   📍 학사-교양/기타과목 > 인문사회과학/자연과학기반   [이 위치로 이동]
   ```
5. [이 위치로 이동] 클릭 → 확장앱이 해당 메뉴·영역 탭을 자동 클릭 → 학생은 평소처럼 "책가방 담기" 버튼을 누른다.

## 4. Phase 0 — 사전 기술 조사 (대부분 완료 ✅)

> **2026-07-17 실제 캡처+curl 재현으로 핵심을 확인했다. 상세 근거는 [docs/api-notes.md](docs/api-notes.md).**
> 아래 조사 항목 중 굵게 표시한 것들이 확정되었고, 그 결과 §5의 전략을 전면 재작성했다.

### 확정된 사실 (설계를 바꾼 3가지)
1. **GLS = Nexacro(GAIA) 앱.** 통신은 `POST /gaia/<화면ID>/<메서드>.do`. 예: 전자시간표 과목조회 = `NHSSU030540M/selectMain03.do`.
2. **요청 본문은 클라이언트 암호화 필수** (`Crypto::<base64>::<hex>::<hex>`). 평문 요청은 `ErrorCode=-1`로 거부됨(재현 완료). → **확장앱이 검색어를 바꿔 raw HTTP 쿼리를 날리는 것은 불가능.**
3. **응답은 평문 SSV** (구분자 `\x1e`/`\x1f`). 복호화 불필요. 한 과목 행에 **`INFORM` 필드**로 `영역구분: 글로벌[2020학번이후] / 전문영어[2019학번이전]` 처럼 **"어느 영역 탭에 담아야 하는지"가 이미 들어있다.** `ISU_NAME`(교양/전공 계열), `CAMPUS_NM`(캠퍼스)도 같은 행에 존재.

### 남은 조사 항목 (아래 표) — 구현 초기에 확인

| 조사 항목 | 방법 | 알아내야 할 것 |
|---|---|---|
| 전자시간표 데이터 로딩 방식 | 각 메뉴/영역 탭 클릭 시 Network 탭 관찰 | XHR/fetch 엔드포인트 URL, HTTP 메서드, 요청 파라미터(년도학기 코드, 캠퍼스 코드, 영역 코드), 응답 포맷(JSON/XML) |
| "전체과목 검색" 메뉴의 API | 전체과목 검색에서 키워드 검색 후 Network 관찰 | 응답에 **이수구분/영역 코드 필드가 포함되는지** — 포함된다면 v1이 매우 단순해짐 (아래 전략 A) |
| 인증 방식 | 요청 헤더/쿠키 확인 | 세션 쿠키만으로 호출 가능한지, CSRF 토큰이 필요한지 |
| 페이지 렌더링 구조 | Elements 탭 | 일반 HTML인지, iframe 안인지, Nexacro 같은 UI 프레임워크인지(이 경우 DOM 조작 방식이 달라짐) |
| 메뉴 이동 자동화 가능성 | 메뉴 클릭 시 URL/해시 변화, 호출되는 JS 함수 확인 | 프로그래매틱하게 특정 영역 탭을 열 수 있는지 |
| 영역 코드 ↔ 메뉴 이름 매핑 | 각 영역 탭 요청 파라미터 수집 | 정적 매핑 테이블 작성 (예: `code=03` → "글로벌") |
| 학번별 분류 차이 | INFORM 필드에 이미 학번 분기 텍스트가 있음(확인됨). 검색 요청에 학번 파라미터가 있는지만 추가 확인 | 2025/2020 학번 전·후 분류를 구분해 보여줄 수 있는지 |

## 5. 핵심 설계 — 응답 관찰(패시브 인덱싱) 방식

> **왜 바뀌었나:** 원래는 "확장앱이 검색 API를 직접 호출"(전략 A)하려 했으나, Phase 0에서 **요청 본문이 클라이언트 암호화 필수**임이 확인됐다. 확장앱이 임의 검색어로 raw 요청을 만들려면 Nexacro의 암호화 로직을 복제해야 하는데 이는 취약하고 유지보수가 어렵다. 반면 **응답은 평문 SSV**이고 필요한 영역 정보(`INFORM`)가 전부 들어있다. → **"요청을 만들지 말고, 페이지가 이미 주고받는 응답을 가로채 인덱싱한다"** 가 가장 견고한 길이다.

### 전략 A′ (채택): 패시브 응답 인덱싱 + 로컬 검색
1. 확장앱이 페이지의 XHR/`fetch`를 MAIN world에서 후킹한다(응답 읽기 전용).
2. 응답이 `Dataset:dsGrdMain03`(또는 동종 과목 데이터셋)을 포함하는 SSV면, 파싱해 과목 행을 추출한다 — 각 행에 `INFORM`(영역), `ISU_NAME`(계열), `CAMPUS_NM`(캠퍼스)이 이미 있다.
3. 추출한 과목을 `년도학기` 단위 로컬 인덱스(`chrome.storage.local` → 초과 시 IndexedDB)에 누적 저장.
4. 사용자가 확장앱 검색바에 키워드 입력 → **전부 로컬에서** 검색(즉시, 부분일치·초성·오타허용 가능).
5. 결과 카드에 과목 정보 + **영역 경로**(예: `학사-교양/기타과목 > 글로벌`)를 표시.

- **장점:** 암호화 복제·복호화 불필요, 응답 SSV 포맷(안정적)에만 의존, 서버에 추가 부하 없음(사용자가 원래 하는 통신만 관찰).
- **한계:** 인덱스는 "사용자가 한 번이라도 응답을 받아본 과목"만 포함 → 최초에 데이터를 모으는 방법이 필요(아래 B′).

### 전략 B′ (보완): 능동 수집 — 페이지의 Nexacro 조회를 대신 트리거
INFORM만으로 전 영역을 커버하려면 각 영역/캠퍼스 목록을 한 번씩 받아와야 한다. 이를 위해:
- **선호 방식:** MAIN world에서 `nexacro.getApplication()`으로 앱 객체에 접근 → 전자시간표 폼의 "조회" 트랜잭션을 영역/캠퍼스를 바꿔가며 프로그래매틱 호출(암호화는 Nexacro가 알아서 처리). 응답은 위 패시브 훅이 그대로 인덱싱.
- **대체 방식:** 자동 호출이 어려우면, 사용자가 "이번 학기 전체 수집" 버튼을 누르고 확장앱 안내에 따라 영역 탭을 한 번씩 클릭 → 그동안 패시브 훅이 전부 모음(학기당 1회, 수십 초).
- 캐시 키에 `년도학기` 포함 → 학기 바뀌면 재수집 유도, 수동 갱신 버튼 제공.
- 결과 카드에 "수집 시각" 표시로 최신성 한계 보완.

> **B′의 능동 트리거(`nexacro` 접근)는 §4 남은 조사 #3에서 가능 여부를 먼저 확인한다.** 불가하면 패시브 누적(사용자 클릭 기반)만으로도 기능은 성립한다.

## 6. 아키텍처 (Chrome/Whale Extension, Manifest V3)

```
┌─ GLS 전자시간표 페이지 (Nexacro) ───────────────────────┐
│  [MAIN world] 주입 스크립트 (page-hook.js)              │
│   ├─ window.fetch / XMLHttpRequest 후킹 (응답 읽기 전용)  │
│   ├─ SSV 응답에서 dsGrdMain03 파싱 → 과목행 추출          │
│   ├─ (선택) nexacro.getApplication() 로 조회 트리거/탭이동 │
│   └─ window.postMessage 로 아래 content script에 전달     │
│                                                        │
│  [ISOLATED world] content script                       │
│   ├─ 검색바 UI 주입 (Shadow DOM 로 스타일 격리)           │
│   ├─ postMessage 수신 ↔ background 중계                  │
│   └─ 검색 결과 렌더링 / "이 위치로 이동" 요청              │
└───────────────┬────────────────────────────────────────┘
                │ chrome.runtime 메시지
┌───────────────┴────────────────────────────────────────┐
│  service worker (background)                            │
│   ├─ 인덱스 저장/조회 (chrome.storage.local / IndexedDB)  │
│   └─ 학기별 캐시 관리, 검색 랭킹                          │
└─────────────────────────────────────────────────────────┘
   + popup: 확장앱 아이콘 → 캐시 상태 / [전체 수집] / 설정
```

- **왜 MAIN world 훅이 핵심인가:** MV3 content script는 isolated world라 페이지의 `fetch`/`nexacro`에 직접 접근할 수 없다. `world: "MAIN"`(또는 `<script>` 주입)으로 페이지 컨텍스트에 들어가야 응답을 가로채고 Nexacro 앱을 조작할 수 있다. 두 world는 `window.postMessage`로 통신한다.
- **인증은 신경 쓸 필요 없음:** 우리는 요청을 새로 만들지 않고 페이지가 이미 인증된 상태로 받는 응답을 읽을 뿐 → 세션/쿠키/암호화를 다룰 필요가 없다.
- UI는 Shadow DOM 안에 렌더링해 GLS CSS와 충돌 방지.
- GLS가 iframe/프레임 구조면 `all_frames: true`로 주입하고 대상 프레임을 URL로 식별.

### SSV 파서 (핵심 유틸, 근거: api-notes.md §4)
- 섹션/레코드 구분 `\x1e`, 필드 구분 `\x1f`.
- `Dataset:<name>` 다음 레코드가 `컬럼명:타입(길이)` 정의, 이후 레코드가 데이터행(첫 필드 `_RowType_`).
- 관심 데이터셋명(`dsGrdMain03` 등)은 상수 목록으로 두고, 스키마가 바뀌면 컬럼명 기반으로 매핑(위치 의존 최소화).

### INFORM 파싱 규칙 (실제 예시 2건으로 확정)
`INFORM` 원문 예:
- `**영역구분: 글로벌[2020학번이후] /  전문영어[2019학번이전]`
- `**영역구분: 인문사회과학/자연과학기반[2020학번이후] /  기초자연과학[2019학번이전]`

파싱 절차:
1. 앞의 `**영역구분:` 라벨과 좌우 공백 제거.
2. 정규식 `/([^\[\]]*?)\s*\[([^\]]+)\]/g` 로 `(영역명, 학번조건)` 쌍을 **모두** 추출. 영역명 캡처의 앞쪽 잔여 `/`·공백은 트림.
3. 결과: `[{area:"인문사회과학/자연과학기반", cohort:"2020학번이후"}, {area:"기초자연과학", cohort:"2019학번이전"}]`

> ⚠️ **영역명 안에 `/`가 있으므로**(`인문사회과학/자연과학기반`) 절대 바 `/` 로 split 하지 말 것. 반드시 대괄호(`[학번]`)를 구분 기준으로 삼는다. 학번 breakpoint는 데이터마다 다를 수 있으니(2020/2019 관측, 화면 경고문엔 2025 언급) 대괄호 안 문자열을 그대로 보존한다.

### 데이터 모델 (로컬 인덱스)

```ts
interface Course {
  yearTerm: string;        // "2026-15"  (GAESUL_YEAR + GAESUL_TERM)
  campus: string;          // CAMPUS_NM  예: "자연과학"
  isuType: string;         // ISU_NAME   예: "교양"  (좌측 메뉴 계열 힌트)
  courseCode: string;      // HAKSU_NO   예: "GEDG006"
  section: string;         // BUNBAN     예: "41"
  name: string;            // GWAMOK_NAME 예: "과학영어"
  nameEng?: string;        // GWAMOK_ENG_NAME
  professor: string;       // PER_NAME
  credits: string;         // HAKJUM     예: "2(3)"
  schedule?: string;       // GYOSI_NAME
  areaRaw: string;         // INFORM 원문 (영역구분 텍스트)
  areas: AreaTag[];        // INFORM 파싱 결과 (아래)
  fetchedAt: number;
}

// INFORM 예: "**영역구분: 글로벌[2020학번이후] /  전문영어[2019학번이전]"
interface AreaTag {
  area: string;            // "글로벌"
  cohort?: string;         // "2020학번이후"  (없을 수도)
}

interface IndexMeta {
  yearTerm: string;
  totalCourses: number;
  collectedAt: number;
  coverageNote?: string;   // 패시브 누적이면 "부분 수집" 등 최신성 표시
}
```

### 검색 로직
- 정규화: 공백 제거, 소문자화, 로마숫자/아라비아숫자 통일("일반물리학2" ↔ "일반물리학II").
- 매칭 우선순위: 학수번호 완전일치 > 과목명 완전일치 > 과목명 부분일치 > 초성일치(v2).
- 동명 과목이 여러 분반/캠퍼스/영역에 걸쳐 있으면 **전부 나열**하고 영역별로 그룹핑해 표시.

## 7. UI 설계

### 페이지 내 검색바 (주 인터페이스)
- 전자시간표 화면 상단에 고정 배치, 단축키(예: `Ctrl+K`)로 포커스.
- 입력 시 디바운스(300ms) 후 검색, 결과는 드롭다운 카드 목록.

### 결과 카드
```
┌─────────────────────────────────────────────┐
│ 일반물리학2  GEDB005-41                       │
│ 홍길동 · 3학점 · 자연과학캠퍼스 · 월수 10:30    │
│ 📍 학사-교양/기타과목 > 인문사회과학/자연과학기반 │
│                          [이 위치로 이동 →]   │
└─────────────────────────────────────────────┘
```
- 학번 전/후 분류가 다른 경우 두 경로 모두 표시하고 배지로 구분 → **`INFORM` 필드에 이미 학번 분기가 텍스트로 들어있어 구현 가능함이 확인됨** (예: 글로벌[2020학번이후] / 전문영어[2019학번이전]).
- "이 위치로 이동": Nexacro 앱 접근이 가능하면 전자시간표 조회 트랜잭션을 해당 영역/캠퍼스로 트리거해 그 탭 상태로 만든다. 불가하면 경로 안내 + "여기 클릭하세요" 하이라이트로 대체(fallback).

### popup (보조)
- 캐시 상태(학기, 과목 수, 수집 시각), [데이터 갱신] 버튼, 캠퍼스 기본값 설정.

## 8. 기술 스택

| 항목 | 선택 | 이유 |
|---|---|---|
| 플랫폼 | Chrome Extension Manifest V3 | Chrome/Edge/Whale(수강신청에 많이 쓰는 브라우저) 모두 Chromium 기반 |
| 언어 | TypeScript | API 응답 스키마를 타입으로 고정해 사이트 변경 시 파악 용이 |
| 번들러 | Vite + @crxjs/vite-plugin | HMR 지원, MV3 설정 간편 |
| UI | Preact (또는 vanilla + lit-html) | 콘텐츠 스크립트에 주입하기에 가벼움 |
| 저장소 | chrome.storage.local → 초과 시 IndexedDB | 한 학기 전체 과목(수천 건)도 수 MB 수준 |
| 테스트 | Vitest + 캡처한 응답 fixture | 실서버 없이 파서/매핑 로직 검증 |
